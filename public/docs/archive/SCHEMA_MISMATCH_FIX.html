Schema Mismatch Fix - Production Database Compatibility

Branch: patch/fix-schema-mismatch-and-crud
Date: 2025-11-13
Status: ‚úÖ Fixed and Ready for Testing

Overview

This patch fixes a critical schema mismatch issue where the code was trying to use database columns that don‚Äôt exist in the production database yet.

Problem

Users were unable to submit feedback, receiving the error:

POST https://fbmtayllfxudwfqcazgu.supabase.co/rest/v1/feedback 400 (Bad Request)
Could not find the 'user_email' column of 'feedback' in the schema cache

Root Cause

The FeedbackModal.tsx component was trying to insert data into user_email and user_name columns that exist in the migration file (20251112000001_create_feedback_table.sql) but haven‚Äôt been applied to the production database yet.

Code was trying to insert:

{
    user_id: user?.id,
    user_email: user?.email || "anonymous@example.com",  // ‚ùå Column doesn't exist in prod
    user_name: user?.display_name || "Anonymous User",   // ‚ùå Column doesn't exist in prod
    type,
    title: title.trim(),
    description: description.trim(),
    category: category.id,
    status: "open",
}

Production database schema only has: - id - user_id - type - title - description - category - status - created_at - updated_at

Solution

File: FeedbackModal.tsx

Removed the user_email and user_name fields from the insert operation to match the production database schema:

const { error: insertError } = await supabase
    .from("feedback" as any)
    .insert({
        user_id: user?.id,
        type,
        title: title.trim(),
        description: description.trim(),
        category: category.id,
        status: "open",
        // Note: user_email and user_name columns don't exist in production yet
        // priority and status have DEFAULT values in the database
    });

Why This Approach?

  1. Backward Compatible: Works with both old and new database schemas
  2. No Migration Required: Can be deployed immediately without database changes
  3. User Info Preserved: The user_id foreign key still links to the users table, so we can get email/name via JOIN if needed
  4. Future-Proof: When the migration is eventually applied, we can add these fields back
